Notes

  1. If you are using Cash Profiling and you wish the standard costs of the time incurred to be treated as cash, enter Y here. Default = N.
  2. The feature allows you to retire activities, and make them no longer available.

You can retire these activities on a specific date, e.g. your year-end 31/12/2006.

This parameter allows the old “retired” activity to be used for entries up to that date – less the days specified here.

  1. There are 2 parameters which are linked.

If you have any kind of open book contract then the recommended settings should be TSCOSTM = N, TSDET=Y.

TSCOSTM: N – default means one entry in JC per employee/activity/day/analysis, i.e. 3 entries in 1 day for this combination will produce 3 cost postings. Y-means that costs will be summarised for the period you choose when you “Create Timesheet Batch”.

TSDET. Is a further layer of detail on the above – “Y” means create 1 cost posting for every entry in the timesheet

  1. Timesheet employees are separate lists to those in system user logins (and separate to the salaries system) to allow you to operate timesheets as a central function.

You can link system user logins and timesheet employees by setting this to “Y”.

Unless you have a central time entry function set this to “Y”.

This will assume that individuals will enter their own timesheets.

  1. N=enter time is decimal places of hours
  2. Daily Hours validation: Hours in any 1 day should be >= TSHRSDAY and <= TSMAXHRS.

TSUNIT ensures that people are not entering e.g. 0.01 of 1 hour, and suggests minimum units of time.†

  1. There are 2 input methods.

Daily timesheets or Weekly.

If you are using the latter and it is contract only timesheets (e.g. if you have overhead contracts) set to Y.

  1. A control feature to prevent entry of time to jobs unless activity rates have been set up for them, e.g. where you will be agreeing specific standard costs per job
  2. Not normally used – consult coins as 3rd Party software may be required
  3. Transaction type in JC/GL (Must exist as a valid transaction type).
  4. Can be varied, typically this will be either Friday (6) or Sunday (1).
  5. Self-explanatory
  6. Whether to batch only approved timesheets (Y or N)
  7. Whether to combine timesheet postings for employees into a single cost transaction per analysis with 0 value (Y or N). If so, a second cost transaction is created for all employees per analysis with the combined values.

This allows for "double posting", where the timesheet posting program generates a debit costing line with employee name and hours only. When the transactions are posted, the contract costing lines are generated one line per employee with name and detail of the days worked, and with days in the Quantity field of the costing transaction, but with zero cost. An additional transaction is generated to each cost code used, with an aggregated description, such as 6 Employees, and the total aggregated cost.

  1. The days of the week that are considered non-working. The default is 1,7 which is Sunday and Saturday
  2. A can-do list(a list in which wildcards are used to omit or include certain values. For example, !AB*,* means include all codes except those beginning "AB". of contract groups that are treated as overhead contracts.)

If a timesheet line is entered for a contract whose contract group is in this list AND the employee is an overhead employee (that is, the Non-costed field in Contract Status > Timesheets > Employees is ticked), no cost transaction is created for that timesheet line when the timesheets are posted.

However, if overhead category staff are allocated to a mainstream contract, or normal costed staff are allocated to an overhead contract, the costing to the contract takes place.
