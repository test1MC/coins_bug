History Recording

History is recorded for certain processing in the Workbench or in the Administration menu. This section gives you more details about recording and producing history records in both those areas.

Workbench

Create /Update Header

If new parameter CV/CVRAUDIT is set to Y, the Header creates or updates history records each time you change the Header fields. All fields are copied to the history record.

  • A new history record is created if:
  • A is created;
  • The Status is changed (such as to Locked, or marked as Current, Latest, Complete, Authored, or Unauthored);
  • Other fields are updated and the last history record exists with a different user ID or Date.
  • A history record is updated if fields are updated and the last history record exists with the same user and date.

If a has been copied or created with the Copy previous adjustment option, the source number is reflected in the event description.

Delete

The deletion of a is validated and only allowed when all integrity checks have been passed. All history records tied to the deleted are also deleted.

Mark as Current/Latest

The Mark as Current and Mark as Latest actions on the s screen cause a history record to either be created or updated, depending on the conditions set forth in Section 1.1.1.

Lock/Unlock

When the Locked flag on the Incomplete screen (see Figure 1) is ed (locked) or uned (unlocked), a history record is either created or updated, depending on the conditions set forth in Section 1.1.1.

Locked Flag

Mark as Complete/Incomplete

The Mark as Complete and Mark as Incomplete actions on the Workbench screens cause a history record to either be created or updated, depending on the conditions set forth in Section 1.1.1.

Authore/UnAuthore

The Mark as UnAuthored action on the Main Workbench screen causes a history record to either be created or updated, depending on the conditions set forth in Section 1.1.1.

Release for Authoration

The Release for Authoration workflow action on the Workbench screens is recorded in History. The status of the fields in the history record are assigned in the same way as set forth in Section 1.1.1.

Update Items

If parameter CV/CVRAUDIT is set to Y, the Update Items action on the Workbench screens results in the recording of the updated event in a history record, as follows:

  • Fields in the history record are assigned in the same way as set forth in Section 1.1.1.
  • The Description is set to Items Updated in the History browse.
  • A list of processed (updated) Items are recorded as history items.

Generate Spreadsheet/Upload Spreadsheet

If parameter CV/CVRAUDIT is set to Y, Generate Spreadsheet and Upload Spreadsheet actions on the Workbench screens record the event in history, as follows:

  • Fields in the history record are assigned in the same way as set forth in Section 1.1.1.
  • The Description is set to Generated Spreadsheet or Uploaded Spreadsheet in the History browse.
  • An XML spreadsheet file name is recorded in the Spreadsheet File Name field in the History browse.

Post Items

Posting Items includes the Create GL Batch and Post Budgets and Memo Data actions on the Workbench screens. If parameter CV/CVRAUDIT is set to Y, the process of posting Items results in the recording of any event in History, as follows:

  • New history records are generated for each posting, and fields are assigned in the same way as set forth in Section 1.1.1.
  • The Description is created by using one of the Action Types (Create GL Batch with Reversing/Non-reversing option, Post Budgets, or Post to Store Data).
  • A list of processed (posted) Items are recorded as history items.
  • GL batch posting also stores a reverse flag and Batch number values in history record fields.
