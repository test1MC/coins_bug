Direct Debit Processing - Resolving Problems

The following sections identify potential problems with advice on how to resolve them.

Direct Debit Data Created Incorrectly

If the direct debit data process has been run from Statement Print but is still at a status of To Email, then the process can be rerun multiple times.

If the process was run in error or for a specific customer that should not be processed in the current , the best way to resolve this is to use the Cancel Extract action on the Approval Workbench, which will remove the Direct Debit records. Then amend the direct debit customer records (to or un the Direct Debit flag) and re-run the Statement Print for Direct Debit Records.

Generate Extract Incorrect

If Generate Extract has been run and the record status is still at To Approve, you can run the Cancel Extract action to remove all Direct Debit records, and start the whole process once again.

This needs to be done with care: if the created file has been approved by the bank, you need to make sure that the banking software has some controls to prevent loaded data for the current from being re-processed (by using some unique reference).

Direct Debit Cash Batch Incorrect

If the Approve Extract action has been run and a batch has been created, then the following process can be followed:

Parameter SL/DDPOST=N

If the cash batch exists and has not been processed yet, you can cancel (delete) the batch in Cash Entry - Batches, which resets the Direct Debit record for the to a status of To Approve and clears the batch number.

Parameter SL/DDPOST=Y or Batch Posted

If the cash batch exists and has been posted, you need to use Cancel Posted Cash to cancel the batch, then use the Reset Status action to reset the Direct Debit record for the to a status of To Approve and clear the batch number.

Batch Status in Posting

If the Direct Debit generated batch is stuck in a posting state, you need to go to the Approval Workbench and click the Batch Number to view the Batch Detail. View the transactions for the batch to find which transactions have a status of not posted. Review these transactions to determine why they were not posted - you can use the Report Status Log file to help identify the issues. Once this is resolved you should be able to post the batch.

If you cannot post the batch because a particular transaction for a customer has an issue, you need to identify and correct the issue and ideally not delete the transaction in error, because if it is deleted the invoice will be reset to unpaid and will be picked up to be re-processed in the following month. The only way to resolve such a situation is to amend the customer so they are not processed via Direct Debit and then create a cash batch manually and enter the receipt for the invoice in question (this assumes that the issue stopping the transaction previously has now been resolved).
