Manually override codes on certain transactions

It may be necessary to manually override the default code on certain transactions following the rate change. Particular attention should be paid to subcontract certificates and transactions as detailed below.

(For Ireland?) Since most construction operations are not standard rated, it is anticipated that most Subcontract Ledger and Ledger transactions will be unaffected. However, clients should consider whether there could be any unpaid standard rated transactions in either ledger and take appropriate action as explained below.

Subcontract Certificate Processing

Note that if a default code is set on the subcontract record, each subcontract certificate will default to that code (although this can be overridden by the user). Thus changing the default on the subcontract records using the utility (see Change Default Codes) will ensure that future certificates will default to the new rate.

However, if there is no default set on the subcontract record, each subcontract certificate will default to the code on the previous certificate for that subcontract. In this case, users will need to override the code on the first certificate for each subcontract after the rate change.

We believe that most clients set a default code on every subcontract record so the default code update utility should be effective in ensuring that future transactions are at the new rate. However, users should be advised to take extra care in checking transactions following the rate change.

Certificate Processing

The processing of transactions is consistent with Subcontract Ledger transactions (see above) as follows. If a default code is set on the series, each transaction will default to that code (although this can be overridden by the user). Thus changing the default on the series records using the utility (see Change Default Codes) will ensure that future transactions will default to the new rate.

However, if there is no default set on the series record, each transaction will default to the code on the previous transaction for that series. In this case, users will need to override the code on the first transaction for each series after the rate change.

We believe that many clients do not set a default code on every subcontract record so the default code update utility may not be effective in ensuring that future transactions are at the new rate. Therefore clients would be advised to put procedures in place for checking the coding of these transactions before the batches are posted.

If you are using separate codes for Sales and Purchases (as has become common practice on newer implementations) then you can retire the old code for sales to prevent errors once you have finished invoicing at the old rate (it can then be un-retired temporarily if you need to post a few transactions with the old rate). However, if you are using the same codes for both Sales and Purchases you will not be able to retire the old code while you are still processing purchase invoices dated before the rate change.
