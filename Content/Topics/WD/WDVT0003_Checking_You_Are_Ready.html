Checking You Are Ready to Create a Return

Ensure all transactions have been posted

Firstly you need to ensure that all relevant transactions have been entered.

You can then use

  • Batch Status Report (single ) or 
  • Batch Status Report (Multi-) 

to confirm that all batches have been posted.

Run Reconciliation Report

The Reconciliation Report can be used before creating a Return to check that the records in reconcile to the control accounts in the General Ledger.  

The report runs for a single registration and selects the and/or sectors linked to that registration - for both the transactions and the GL balances to include.

The layout of the report follows a concept similar to a bank reconciliation.  The report reconciles either the latest return on the system (if you Current Period) or the latest return dated on or before the General Ledger period selected. 

The report always includes a summary section.  You can also choose to include sections giving transactional details of various figures from the summary.

transactions with non-recoverable codes are ignored.  The sign convention is as for the General Ledger, so debit GL balances are shown as positive and credit GL balances as negative.  For transactions,   is shown as positive and as negative.

The report is based on the following assumptions:

  • The amounts which have been returned are transferred out of the  control accounts in the period following that in which the return ends.
  • No direct manual postings are made to the control accounts other than the transfer of returned amounts as detailed above.
  • If the business makes returns to more than one jurisdiction (country) from within one then separate  codes and separate GL control accounts are used for each jurisdiction.

To run the Reconciliation Report:

  1. Go to Returns Workbench.
  2. On the Company Summary tab, select the registration you want to reconcile.
  3. Choose the Reconciliation Report action.

The most likely reason for differences is direct postings to the control accounts.  If this is the case, we recommend you reverse out such postings and re-input them as journals (with a code analysis rather than being coded to the control accounts).  To prevent this happening in future, the control accounts should be set not to allow manual postings.  This can be done either in Account Maintenance or in GL Elements, depending on the settings in Configuration.

Review unreturned transactions

You should check for any exceptional transactions before creating the Return.

To review unreturned transactions:

  1. Go to Returns Workbench
  2. Click the Unreturned Transactions tab.
