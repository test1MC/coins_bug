Removing Users

When you no longer want a user to have access to (for example, when they leave the company), you should disable their User ID.  However, you should not delete their user record; has the policy of retaining details of all historical users in the database for the following reasons:

  • Ensuring that the transactional audit trail retains details of who input and/or authored transactions, orders, payment runs, etc;
  • Ensuring that reprinted documents, including purchase orders, etc. will correctly print without missing information;
  • Ensuring that there is referential integrity within the database where database records refer to user IDs;
  • Preventing problems when users need to edit records which include reference to a user who has left (in some circumstances, if the details of the original user have been deleted then an error message will be displayed and the user will need to overwrite the original details with a current user): this can be both frustrating at the time and misleading for the future;
  • Allowing access to the User Logging to review functions used on by the historical user (subject to the time limit set in SY parameter LOGHIST).

To disable a user in a secure manner:

  1. Un the Named User box in the Named User group of fields on the User Details screen.  This will remove the user from the user count for licensing purposes and will disable their user ID so that they cannot log in.
  2. As added security, various other settings on the user record could be changed as follows to further deny the possibility of them gaining any access (although none of these should be necessary if they are no longer recorded as a named user):
    1. All the groups could be removed from the user -- thus removing their function access and a Group called "" could then be added to their user record as their prime group -- this group would be configured to have no function access;
    2. The user's password could be changed;
    3. The Domain\User field could be blanked (if this is used);
    4. The Restricted IP field could be set (preventing login from any other IP address);
    5. Delete the User Views for this user (see User View Set Up).
  3. You may want to consider other places in where the user is referred to.  For example:
    • If the user is set up in the Sales Staff list in House Sales then this record should be marked as dormant.
    • If the user is set up in the Buyers list in Procurement, this record should be marked as retired.
