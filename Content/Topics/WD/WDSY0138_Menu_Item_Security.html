Menu Item Security

Menu Item Security is an optional way of configuring function security. The main features of this method are:

  • Access can be set at the level of menu items; all individual functions (tabs, buttons, options) within the menu item inherit the permissions from the menu item.
  • Access to menus is automatically granted if the user has access to any item on the menu.
  • Users can be granted "read only" or "update" access to a function.

The benefits of this approach include:

  • Less maintenance: setting permissions at the level of menu items means there are far fewer functions you need to consider.
  • When new features are added to existing menu items they will automatically inherit the permissions from the menu item, so you will not need to set the permissions for them. This greatly reduces the effort in assessing new functions at each release.
  • You can give different types of users "read only" and "update" access to the same menu item, without needing to worry about setting permissions for individual buttons and update functions.
  • You can prevent access to selected sub-functions (such as delete or bulk-update).

Menu Item Security is optional; if you choose not to turn it on, function access security will work as before. If you do turn Menu Item Security on, previous methods of setting access permissions are still available and can be used to fine-tune the permissions if necessary.

How Menu Item Security Works

Every function in the system is associated with a parent menu function (see Parent and Child Functions), and has an access type indicator to show whether it gives update access or read only access. Menu Item Security allows you to set access permissions for functions in OA based on the menu item they belong to, and to grant either read only or update access for all the options that fall underneath that menu option. This is much simpler and quicker than setting permission for individual functions.

Menus, Tabs and Containers

When using Menu Item Security, you do not need to set security for Menu, Tab and Container functions. They will be made available if they contain anything to which the user has access. However, the user will not automatically have access to the other contained functions.

For example, if a user has access to the (%WPL5100BAVM), they automatically have access to the PL menu, which contains it, but they do not automatically have access to the Invoices or the Payments which are also on the PL menu.

Similarly, if a user has access to the My tab which belongs to the My container, they automatically have access to My from the menu, but they do not automatically have access to the other functions (My , My Companies and so on).

However, if a user has explicitly been granted access to a container or tab function (such as My ), they automatically have access to all the functions it contains (this only applies to container and tab functions, not to menus).

Access Type

Each function has an access type indicator: either Read Only or Update. This is determined by so that any function that updates the application database will be identified as an update. (Note: all functions update system logs etc. and these updates are ignored for this purpose.)

For example, a browse screen would be Read Only access, as would the function that enables the export button on that browse, but the functions that enable the add and update buttons would be Update access.

If Menu Item Security is switched on, users are able to set security at the menu item level, and the access options will change from Group/Yes/No to Group/Yes/Yes-Update/No. In this way, a user who is given Yes (that is, read-only) access to a parent function will only be allowed access to the child functions whose access type is Read Only.

NOTE

Existing Access=Y will be considered to be Read Only. If Menu Item Security is switched on, you will need to consider your existing groups carefully.
