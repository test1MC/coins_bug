PL/PCSTLIST - Txn Types For Inv Match/Prov Costing

The transaction types to use in Provisional Costing with Invoice Matching. This provides the ability to use provisional costing combined with GRN matching so that invoices can be retrospectively matched to GRNs and costing altered likewise if needed.

Enter a comma-separated list of two transaction types: the first as the PCST equivalent of any debit transaction types, the second as the PCST equivalent of any credit transaction types).

If this parameter is not set, none of the Provisional Costing with Invoice Matching functionality will work.

The PL parameter DISTTYPE must exclude (R)egistering, but allow both (C)osting and (P)rovisional Costing.

On F1 during invoice matching, if the invoice cannot be fully reconciled, the system presents the opportunity to provisionally cost the invoice and swap its transaction type to the relevant entry in PCSTLIST. This option is only presented on first entry of the invoice. The original transaction type is remembered in order to re-instate that transaction type once the invoice is fully reconciled via the invoice matching routine. It is this final reconciliation that reverses out the provisional costing records and creates new costing records. It should be understood that the setting of the PCSTLIST parameter is vital to the correct reporting of the provisional cost transactions.
