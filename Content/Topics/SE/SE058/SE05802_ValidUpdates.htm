Validation Updates

Since changes to the Inception Date are carried to all appropriate fields throughout the , certain validation processes have been updated to avoid errors that could occur. These validation processes are described below.

Amount - and First Billing Date

The goal of this validation update is to retain the original date order, but still pass all required validation.

The and First Billing Date are changed for all Location/Service Department records where the equals the old Inception Date (there could be more than one, since the can reference multiple Locations/Service Departments).

If the has subsequent Amount record(s) on file for a Location/Service Department with a an that will conflict with the change made to the first record, any subsequent records are also adjusted. All subsequent records are moved one day forward until there are no conflicts in the date.

Billing Schedule – Scheduled Bill Date

There is no validation on the Scheduled Bill Date referenced within the Billing Schedule. You must adjust any data.

Revenue Schedule –

There is no validation on the referenced within the Revenue Schedule. You must adjust any data.

Renewals – Start Date

The Start Date is changed for the first Renewal record where the Start Date equals the old Inception Date. You must change the End Date.

NOTES:

  • Any Planned Inspections and Covered Asset Inspections with a date outside the range of the Renewal Period are removed. You must adjust the tasking data.
  • The Cost Schedule and Cost Budgets contain validation to prevent non-zero values for out of the Renewal date range. You must remove any values covered by the new Inception Date prior to changing the Inception Date.

Term – Start Date

The Start Date is changed for the first Term record where the Start Date equals the old Inception Date. The End Date is automatically recalculated when the Start Date is changed, and the Duration remains unchanged.

If the resulting dates overlap other Term records, you must adjust the other records to provide no overlap in coverage dates.

If the has subsequent Type record(s) on file with a an that will conflict with the change to the first record, any subsequent records are also adjusted. All subsequent records are moved one day forward until there are no conflicts in the date.
