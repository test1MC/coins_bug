Additional node details:

 

Node

Information

Detail/LevelInterest

This corresponds to the level of interest in and is how a user ranks the prospect seriousness about buying.  This is a validated field and will require a value to be specified.  Depending on the users configuration of this can have a value of either N, V or S.  Alternatively the user can specify there own level of interest codes.

Detail/Status

This is the prospect status (e.g first time buyer, house to sell, house sold etc)  This is a validated field in and corresponds to a code in the ‘Purchaser Status’ user defined lookup table in .

Detail/ToReview

This allows you to flag the new prospect as requiring review in .  There is then a separate process for reviewing these new prospects in .

Requirement/MoveStart

Requirement/MoveEnd

When adding a new prospect we prompt for the looking for move timescales as a range of months.  In the above example this would indicate that the purchaser is looking to move in between 1 and 4 months time.

Requirement/MixReq1

Requirement/MixReq2

This is the prospects required market mix (e.g. terraced, semi detached, detached, flat etc).  This is a validated field in and corresponds to a code in the ‘Market Mix’ user defined lookup table in .  These can be left blank.

Requirement/HouseType1

Requirement/HouseType2

The standard house type which a prospect is interested in.  The house type is the style of house.  This would only be used if the user is using standard house types and can be left blank.  You can have development specific house types in , you would specify these in the Event or Development section of the message.

Requirement/LocationReq1

Requirement/LocationReq2

The locations in which a prospect is interested in purchasing.  This corresponds to a user defined code in the contract location table in .  The second location is optional and can be left blank.

CurrentProperty/Mix

 

Similar to Requirement/MixReq1 except it is the market mix of the prospect’s current property.  All CurrentProperty fields are optional.

Event/UserCode

This is the user code for the event that is to be created on the prospect’s history.  This allows a record to be created in the history detailing the initial contact.  Most likely the administrator will create a specific enquiry event for recording contact via their website.

Event/Contract

Event/Plot1

Event/Plot1

Event/HouseType1

Event/HouseType1

The development, plot(s) and house type(s) a prospect has enquired about.  To record these the UserCode event must be configured in to be VISIT or ENQUIRY system event.

Event/AddRequired

Determines whether the development details entered on this event are automatically added to the prospect’s required development list.  If you set to N then these can specified manually using the Developments section of this message.

EnquirySources/Source/Code

This is the code that maps to a valid source of enquiry in .  Most likely a specific source of enquiry will be setup for website enquiries.  Note whether multiple sources of enquiry are allowed is determined by the configuration of the HS module.

 

The source of enquiry is only recorded if UserCode event is configured in to be VISIT or ENQUIRY system event.

MoveReasons/Reason/Code

The reason for buying code.  This maps to a user defined code in the ‘Reason for Buying’ table in .

Developments/Development

This allows you record the development(s), plot(s) and house type(s) a prospect is actually interested in.  These are added to the Prospects Required Development list.

People/Person

Contains the personal details (name, address etc) for the person being attached to the prospect record.

Captions/Caption

This allows you to specify the value for a prospect captions.  These are user defined fields in .  The code and number attributes refer to the unique code given to the caption group and the number is the relative number of the caption in the group (maximum of 10 per group).

 

If the add was successful this would return a XML message with the following format:

 

<?xml version="1.0" encoding="utf-8" ?>

<env:Envelope xmlns:env="http://www.w3.org/2003/05/soap-envelope">

<env:Body>

<call object="hswshvi1" method="addProspect" id="add0001">

<ProspectCompany>001</ProspectCompany>

<ProspectReference>000487</ProspectReference>

<returnvalue></returnvalue>

</call>

</env:Body>

</env:Envelope>

 

This will return the prospect record unique reference (ProspectReference node) in .  Note this reference is only unique within the selected company.

 

If the add failed for any reason the following message would be returned:

 

<?xml version="1.0" encoding="utf-8" ?>

<env:Envelope xmlns:env="http://www.w3.org/2003/05/soap-envelope">

<env:Body>

<env:Fault>

<env:Code>

<env:Value>env:Sender</env:Value>

</env:Code>

<env:Reason>

<env:Text>Person Status does not exist [SY780]</env:Text>

</env:Reason>

</env:Fault>

</env:Body>

</env:Envelope>

 

The env:Text node will contain the reason for failure.
